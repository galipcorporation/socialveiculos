- [x] **B044 — Inconsistência de timezone: `datetime.utcnow()` (naive, deprecado) escrevendo nas mesmas colunas cujo default é tz-aware** · `apps/api/routers/b2b.py` (6×), `apps/api/routers/vitrine_interativo.py` (4×), `apps/api/routers/veiculos.py` (2×), `apps/api/pos_venda_template.py` (1×) · o projeto padroniza datas em **tz-aware** — `models._now()` (`datetime.now(timezone.utc)`) é o `default` de `created_at`/`updated_at`, e praticamente todo o restante da API usa `datetime.now(timezone.utc)`. Efeito: `updated_at` de uma mesma linha alterna entre aware (na criação) e naive (na atualização); além do `DeprecationWarning` do Python 3.12+ (`utcnow()` marcado para remoção), qualquer comparação/ordenação feita **em Python** (não em SQL) que misture os dois estoura `TypeError: can't compare offset-naive and offset-aware datetimes`. Hoje o SQLite mascara o risco (leitura volta naive), mas é frágil e some no Postgres/`timezone=True`. → **Fix:** substituído `datetime.utcnow()` → `datetime.now(timezone.utc)` nos 13 pontos, alinhando as escritas ao mesmo valor aware que o default de coluna já persiste com sucesso (INSERT). Adicionado `timezone` ao import de `datetime` nos 4 arquivos (nenhum o importava). **Verificação:** por ser ambiente degradado nesta rodada (sandbox Linux caiu por falta de disco — sem `pytest`/`py_compile`/git ao vivo), a verificação foi estática: `grep "datetime.utcnow"` no `apps/api` → **0 restantes**; os 4 arquivos passam a importar `timezone`; o valor injetado é **idêntico** ao que `_now()` já grava nessas colunas no INSERT (portanto o SQLite/aiosqlite já o serializa sem erro). ## Padrões a vigiar
Bugs recorrentes neste projeto — checar sempre antes de commitar:

1. **Enums EN vs PT** — código portado do SimuladorFacil usa status em inglês (`approved`, `denied`); os enums/DB daqui são em português (`aprovado`, `negado`). Sempre mapear na fronteira. _(ver B002)_
2. **`settings.api_debug` é minúsculo** — nunca `API_DEBUG`. Atributos de `config.py` são todos snake_case minúsculo. _(ver B001)_
3. **Imports sempre absolutos no `apps/api`** — a pasta NÃO é pacote (sem `__init__.py`); uvicorn roda com cwd=`apps/api`. Usar `from simulador...`, `from routers...`, nunca `..`. _(ver B004)_
4. **Código portado de JS** — cuidado com `Date.now()`, `console.log`, `JSON.stringify` etc. vazando em arquivos `.py`. _(ver B006)_
5. **Imports no topo** — Python não pega `import` faltando em build (só em runtime, na rota chamada). Conferir uso de `os`, `json`, etc. _(ver B005)_
6. **`main.py` só importa router que existe** — antes de adicionar `include_router`, confirmar que o módulo existe; um import quebrado derruba a API inteira. _(ver B008)_
7. **Validar boot após mexer em rotas/models** — `cd apps/api && python -c "import main; print(len(main.app.routes))"` deve rodar limpo. ⚠️ **Atenção (FastAPI ≥0.139):** `len(app.routes)` NÃO reflete mais o nº de endpoints (o `include_router` agora registra 1 wrapper `_IncludedRouter` por router, não cada rota). Para contar/checar endpoints reais use `len(main.app.openapi()["paths"])` ou `"/v1/...": in main.app.openapi()["paths"]`. O boot "rodar sem exceção" continua válido; só o número perdeu o sentido. _(ver B026)_
8. **Nunca usar `except:` nu** — bare except captura `KeyboardInterrupt`/`SystemExit` e mascara sinais de shutdown/erros de programação. Usar sempre `except Exception:` (ou o tipo específico). Os 15 casos herdados no motor do Simulador foram corrigidos na Execução #14 (2026-07-02); checar antes de commitar novo código de automação. _(higiene, E722/B902)_
